## SQLAlchemy 架構

SQLAlchemy 分成兩層：

### Core（底層）
直接用 SQL 表達式語言操作資料庫，接近原生 SQL。

```python
with engine.connect() as conn:
    result = conn.execute(text("SELECT * FROM users"))
```

### ORM（上層）
用 Python class 對應資料庫 table，SQLAlchemy 幫你轉成 SQL。

```python
class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)
    name = Column(String)
```

> **結論**：大多數人用 ORM，但兩者可以混用。Alembic 不一定要搭配 ORM，Core 也可以用。

## Alembic 是什麼

- SQLAlchemy **官方推薦**的資料庫遷移（migration）工具
- 由同一個作者（Michael Bayer）開發
- 功能：版本控制、自動產生遷移腳本、支援升級/降級

## alembic init 產生的檔案

```
your_project/
├── alembic.ini          ← 主要設定檔
└── alembic/
    ├── env.py           ← 核心執行檔
    ├── script.py.mako   ← migration 腳本模板
    └── versions/        ← 存放所有 migration 腳本
```

| 檔案 | 用途 | 需要改嗎？ |
|------|------|-----------|
| `alembic.ini` | 設定資料庫連線字串 | ✅ 通常改 |
| `env.py` | migration 執行的入口點 | ✅ 一定要改 |
| `script.py.mako` | 產生腳本用的模板 | ❌ 通常不動 |
| `versions/` | 所有 migration 腳本 | 自動產生 |

## env.py 詳解

`env.py` 是 **Alembic 和資料庫之間的橋樑**，執行 `alembic upgrade head` 時會跑這個檔案。

### 職責
1. 建立資料庫連線
2. 告訴 Alembic 你的 schema 長什麼樣子（`target_metadata`）
3. 實際執行 migration SQL

### offline vs online 模式

| 模式 | 用途 | 觸發方式 |
|------|------|---------|
| offline | 不連資料庫，只輸出 SQL 文字 | `alembic upgrade head --sql` |
| online | 真正連線並執行 SQL | 一般執行 `alembic upgrade head` |

### 設定資料庫連線（推薦做法）

```python
# env.py 頂部
from app.core.config import settings
config.set_main_option("sqlalchemy.url", str(settings.SQLALCHEMY_DATABASE_URL))
```

> `str()` 是必要的，因為 Pydantic 的 `PostgresDsn` 型別需要轉成字串。

## Sync vs Async

### 生活比喻

| 概念 | 比喻 | 說明 |
|------|------|------|
| **Sync** | 排隊點餐 | 等待期間程式卡住，什麼都不能做 |
| **Async** | 叫號點餐 | 等待期間可以做其他事 |

### 兩個世界的限制

```python
# ❌ sync 函式裡不能直接 await
def sync_function():
    result = await async_function()  # SyntaxError

# ❌ async 函式不能直接被普通呼叫
async_function()  # 只會回傳 coroutine 物件，不會執行

# ✅ sync 啟動 async 的唯一入口
asyncio.run(async_function())

# ✅ async 裡執行 sync
async def async_function():
    await connection.run_sync(sync_function)
```

## async env.py 改寫

### 為什麼要改成 async？

如果專案用 async driver（`asyncpg`、`aiosqlite`），engine 就是 async 的，必須在 async 環境裡使用。

### 執行流程

```
sync 入口
asyncio.run()
  └─ async run_migrations_online()    # 建立 async engine 和連線
       └─ async with engine.connect() as connection  # 建立 async 資料庫連線
            └─ await connection.run_sync(do_run_migrations)  # 橋樑：async → sync
                  └─ context.run_migrations()  # Alembic 實際執行 migration SQL（sync）
```

### 完整程式碼

```python
import asyncio
from sqlalchemy.ext.asyncio import async_engine_from_config
from sqlalchemy import pool
from alembic import context
from app.core.config import settings
from app.models import Base

config.set_main_option("sqlalchemy.url", str(settings.SQLALCHEMY_DATABASE_URL))
target_metadata = Base.metadata

def run_migrations_offline():
    # offline 不涉及真實連線，不需要 async
    url = config.get_main_option("sqlalchemy.url")
    context.configure(url=url, target_metadata=target_metadata,
                      literal_binds=True, dialect_opts={"paramstyle": "named"})
    with context.begin_transaction():
        context.run_migrations()

def do_run_migrations(connection):
    # 同步函式，Alembic 看得懂
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()

async def run_migrations_online():
    connectable = async_engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()

if context.is_offline_mode():
    run_migrations_offline()
else:
    asyncio.run(run_migrations_online())
```

### alembic.ini 連線字串要用 async driver

```ini
# PostgreSQL
sqlalchemy.url = postgresql+asyncpg://user:pass@localhost/dbname

# MySQL
sqlalchemy.url = mysql+aiomysql://user:pass@localhost/dbname

# SQLite
sqlalchemy.url = sqlite+aiosqlite:///test.db
```

## Base.metadata 的運作

### 為什麼只要 `Base.metadata` 就能偵測所有 model？

繼承 `Base` 的瞬間，table 資訊就**自動註冊**到 `Base.metadata` 裡：

```python
class User(Base):  # 繼承的瞬間就登記進去了
    __tablename__ = "users"

class Post(Base):  # 這個也是
    __tablename__ = "posts"

# Base.metadata 裡面已經有兩張表
print(Base.metadata.tables)
# {'users': Table('users'...), 'posts': Table('posts'...)}
```

### ⚠️ 常見陷阱：model 必須被 import 過才會登記

```python
# env.py

# ❌ 只 import Base，但 User、Post 從未被 import → metadata 是空的
from app.models.base import Base

# ✅ 解法：在 models/__init__.py 裡 import 所有 model
from app.models import Base  # 觸發 __init__.py，順便載入所有 model
```

### models/__init__.py 正確寫法

```python
# models/__init__.py
from app.models.base import Base
from app.models.user import User
from app.models.post import Post
# 每次新增 model 都要加進來
```

### import 路徑要統一

從專案根目錄算起，路徑前綴要一致：

```python
# ✅ 正確（從根目錄算）
from app.models.user import User

# ❌ 錯誤（少了 app 前綴）
from models.user import User
```

## 常用指令

```bash
# 初始化
alembic init alembic

# 產生 migration 腳本（不影響資料庫）
alembic revision --autogenerate -m "描述這次改了什麼"

# 執行 migration（真正改變資料庫）
alembic upgrade head

# 回滾一步
alembic downgrade -1

# 查看目前版本
alembic current

# 查看歷史
alembic history

# 只輸出 SQL，不執行（offline 模式）
alembic upgrade head --sql

# 告訴 Alembic 資料庫已在最新版（不重跑）
alembic stamp head
```

### 指令對比

| 指令 | 做了什麼 | 影響資料庫？ |
|------|---------|------------|
| `alembic revision --autogenerate` | 產生 migration 腳本 | ❌ 否 |
| `alembic upgrade head` | 執行 migration 腳本 | ✅ 是 |

## psql 查看 schema

```bash
# 列出所有 table
\dt

# 看某個 table 的結構
\d users

# 列出所有 schema
\dn

# 列出所有 index
\di
```

用 SQL 查：

```sql
-- 看所有 table
SELECT table_name FROM information_schema.tables
WHERE table_schema = 'public';

-- 看某個 table 的欄位
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'users';
```

## 注意事項

### ⚠️ versions/ 資料夾不能隨便刪

Alembic 用版本鏈追蹤資料庫狀態：

```python
revision = 'b2c3d4'
down_revision = 'a1b2c3'  # 指向上一個 migration
```

| 情況 | 後果 |
|------|------|
| 刪掉中間某個檔案 | 版本鏈斷掉，upgrade 報錯 |
| 刪掉所有檔案但資料庫已跑過 | Alembic 找不到對應版本 |
| 資料庫還是空的 | ✅ 可以安全刪掉重來 |

### Pydantic extra fields 錯誤

`.env` 有但 `Settings` 沒定義的欄位會報錯，解法：

```python
class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env", extra="ignore")
```

### 要不要 squash migrations？

大型專案通常**不整理**，原因：
- 合併後舊環境無法正常升級
- 多人開發很難同步

建議做法：用清楚的 `-m` 描述讓檔案好找，只在**開發初期**（production 上線前）才考慮重置。
